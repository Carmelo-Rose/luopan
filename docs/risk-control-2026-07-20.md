# 2026-07-20 风控排查与恢复日志

## 背景

服务机在手动执行大盘采集 `python run.py --multi` 时，20/20 个类目全部采集到 0 个商品。日志中榜单接口持续返回：

- `code=10001010A`：当前请求正在放空，请稍后重试
- `code=11001`：当前网络不稳定，请稍后重试

同时，详情页探测脚本 `probe_risk_control.py` 可以正常取得价格，说明详情页链路和短视频榜列表接口 `video_bring_good` 的风控判定路径不同。

## 初始判断

初始现象看起来像服务机仍处于榜单接口风控冷却中：

- 大盘 20 个类目全部 0 商品，不是单个类目偶发失败。
- API 失败后 DOM 兜底也拿到 0 条，说明旧采集链路无法获得有效榜单数据。
- 详情页正常不代表榜单列表接口正常。

因此当时建议不要继续立即跑服配，避免在同一个接口上继续增加请求。

## 关键排查结论

后续在服务机和本机对比后，实际根因更偏向采集方式触发风控，而不是服务机整体不可用：

- 服务机页面本身可以正常登录并渲染抖音罗盘短视频榜。
- 页面自然操作产生的榜单 XHR 能返回数据。
- 旧代码通过 `page.evaluate(fetch)` 手动构造并请求 `video_bring_good` 接口，容易触发 `10001010A` / `11001`。
- 详情页价格接口与榜单接口属于不同风控路径，详情页正常不能证明榜单接口可用。
- 将采集方式改成页面内真实点击类目、翻页，并等待页面自身 XHR 后，服务机可恢复采集。

## 本次采取的临时修复

调整采集链路，使榜单采集尽量走页面原生交互：

- 类目选择改为点击页面可见的 Aurora cascader 组件。
- 首页数据改为等待页面自然发出的榜单 XHR。
- 后续分页改为点击页面分页按钮，而不是手动 `fetch` 下一页 API。
- 增加 0 有效采集的失败保护，避免空数据继续写表、发摘要或进入后续任务。
- 服务机批处理增加短路逻辑：大盘失败则不继续跑服配，服配失败则不继续执行后续同步。
